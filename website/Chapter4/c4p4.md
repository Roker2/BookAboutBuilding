# Параграф 4: SEPolicy

## Что такое SELinux?

Это опция ядра Linux, которая ограничивает доступ к системным ресурсам. По умолчанию SELinux блокирует все. Чтобы какой-либо сервис имел доступ к ресурсам, пишутся правила - SEPolicy.

## Как работает SELinux?

* Enforcing - SELinux проверяет каждое действие, что происходит в системе. Если SEPolicy разрешает действие, то SELinux дает возможность его произвести. В противном случае, SELinux его блокирует и выводит ошибку в лог, к примеру:
  
  ```
  avc: denied { write } for comm="power@1.2-servi" name="tp_double_tap" dev="proc" ino=4026533160 scontext=u:r:hal_power_default:s0 tcontext=u:object_r:proc:s0 tclass=file permissive=0
  ```
  
  Это называется отказ SELinux (SELinux denial). Такие отказы не дают системе и ее компонентам нормально работать, поэтому Android может работать с багами, нестабильно или вовсе не загрузиться.

* Permessive - SELinux ничего не блокирует, только выводит в лог действия, которые должны быть запрещены. Используется для разработки, особенно полезно на ранних этапах, когда специфичных для устройства правил SEPolicy еще нет.

## Готовые правила в LineageOS

Правила SEPolicy, которые специфичны для AOSP, находятся в platform/system/sepoicy. Производители SoC под свое железо пишут отдельно правила. К примеру, Qualcomm предоставляет их в виде исходного кода через CAF. LOS, в свою очередь, тоже имеет репозиторий с правилами, чтобы код, написанный разработчиками прошивки, работал.

## Как написать правила SEPolicy?

Этот процесс разделяется на несколько ступеней:

* Создание метки (Labeling)
  
  * Сервиса
  * Приложения
  * Файловой системы
  * Пропов (Properties)

* Разрешения

Также для создания меток можно использовать regex.

### Создание метки файла

Обычно метка имеет следующий вид:

```
/path/to/initiator u:object_r:context_name_you_want:s0
```

Для сервиса NFC это может выглядеть так:

```
/(vendor|system/vendor)/bin/hw/android\.hardware\.nfc@1\.2-service\.sec u:object_r:hal_nfc_default_exec:s0
```

Такие правила можно писать и для файлов с папками:

```
/data/vendor/qcam(/.*)? u:object_r:vendor_camera_data_file:s0
```

Такие метки обычно помещаются в `file_context`. [Пример из AOSP](https://android.googlesource.com/platform/system/sepolicy/+/refs/heads/main/private/file_contexts).

### Создание метки приложения

Обычно метка имеет следующий вид:

```
user=user_of_app seinfo=info name=name_of_app domain=scontext_to_assign type=type_of_file
```

В пример возьмем приложение `IWlanService.apk` с именем пакета `vendor.qti.iwlan`. Это приложение известно как приложение `qtidataservices` из состава ПО Qualcomm. Метка будет выглядеть так:

```
user=radio seinfo=platform name=vendor.qti.iwlan domain=qtidataservices_app type=radio_data_file
```

Если в манифесте приложения `AndroidManifest.xml` есть поле `android:process="name"`, то требуется декомпилировать приложение, к примеру, инструментом `apktool` и указать `name` в правиле.

Такие правила помещаются в `seapp_contexts`. [Пример из AOSP](https://android.googlesource.com/platform/system/sepolicy/+/refs/heads/main/private/seapp_contexts).

### Создание метки файловой системы

Обычно метка имеет следующий вид:

```
genfscon filesystem_name partial_path filesystem_context
```

Так может выглядеть метка для раздела `/proc/hwmodel`:

```
genfscon proc /hwmodel u:object_r:proc_fih:s0
```

Такие правила помещаются в `genfs_contexts`. [Пример из AOSP](https://android.googlesource.com/platform/system/sepolicy/+/refs/heads/main/private/genfs_contexts).

### Создание метки пропов

Пропы могут использоваться для контроля поведения каких-либо специфических функций или описания устройства. Такие пропы тоже требуют правил SEPolicy, чтобы получить доступ к ним.

Обычно метка имеет следующий вид:

```
property_name u:object_r:property_type:s0
```

Пример ошибки SELinux, когда у Camera HAL нет доступа к пропу:

```
avc:  denied  { set } for property=camera.tunning.live pid=756 uid=1047 gid=1005 scontext=u:r:hal_camera_default:s0 tcontext=u:object_r:default_prop:s0 tclass=property_service permissive=0
```

Так как этот проп относится к камере, стоит метку назвать как `camera_prop`.

```
camera.tunning.live u:object_r:camera_prop:s0
```

Такие правила помещаются в `property_contexts`. [Пример из AOSP](https://android.googlesource.com/platform/system/sepolicy/+/refs/heads/main/private/property_contexts).

### Комбинирование нескольких меток в одну

Как написано выше, можно использовать regex. Это можно использовать во всех правилах. Также можно несколько меток назвать одним именем, чтобы был одинаковый и одновременный доступ к нескольким сервисам, пропам, файлам и так далее.

```
/sys/class/tp_glove/device(/.*)? u:object_r:proc_touchpanel:s0
/sys/devices/virtual/tp_glove(/.*)? u:object_r:proc_touchpanel:s0
```

```
fpc.fp.miui.token               u:object_r:vendor_fp_prop:s0
goodix.fp.                      u:object_r:vendor_fp_prop:s0
ro.boot.fpsensor                u:object_r:vendor_fp_prop:s0
ro.hardware.fingerprint         u:object_r:vendor_fp_prop:s0
ro.register.count               u:object_r:vendor_fp_prop:s0
ro.template.count               u:object_r:vendor_fp_prop:s0
sys.fp.                         u:object_r:vendor_fp_prop:s0
```

### Создание правила разрешения

Чтобы создать разрешение, требуется следовать этому примеру: `allow scontext tcontext:tclass permission;`.

Пример для конкретного отказа:

```
avc: denied { read write } for pid=4565 comm="init.qcom.post_" name="read_ahead_kb" dev="sysfs" ino=52742 scontext=u:r:qti_init_shell:s0 tcontext=u:object_r:sysfs_dm:s0 tclass=file
```

Получившееся правило:

```
allow qti_init_shell sysfs_dm:file { read write };
```

И это правило положить в `scontext.te`. То есть, правило из примера должно быть в файле `qti_init_shell.te`.

### Макросы

Макросы позволяют упростить и объединить правила. Пример использования макроса `r_file_perms`:

```
allow hal_power_default sysfs:file { read open watch lock };
allow hal_power_default sysfs:file r_file_perms;
```

Все макросы можно найти в AOSP в [global_macros ]([public/global_macros - platform/system/sepolicy - Git at Google](https://android.googlesource.com/platform/system/sepolicy/+/refs/heads/main/public/global_macros))и [te_macros]([public/te_macros - platform/system/sepolicy - Git at Google](https://android.googlesource.com/platform/system/sepolicy/+/refs/heads/main/public/te_macros)).

### Где должны лежать правила?

Начиная с 8.1.0, где появился Project Treble, появилась возможность выбрать, где правила должны быть. Так называются папки sepolicy:

* system: правила для модулей в разделе system;

* vendor: правила для модулей в разделе vendor;

* public: правила, которые доступны модулям из разделов system и vendor;

* private: правила, которые специфичны для платформы, доступны модулям из раздела system;

* dynamic: если нет отдельного раздела vendor, то тогда правила попадут в правила vendor, в противном случае попадут в system.

### Если нет логов SEPolicy.

Откройте defconfig вашего устройства в ядре. Добавьте следующие флаги, если их нет:

```defconfig
CONFIG_AUDIT=y
CONFIG_AUDITSYSCALL=y
CONFIG_HAVE_ARCH_AUDITSYSCALL=y
CONFIG_AUDIT_ARCH_COMPAT_GENERIC=y
```

И пересобирите ядро.
